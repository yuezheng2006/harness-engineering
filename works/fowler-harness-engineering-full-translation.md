# 面向编码智能体用户的 Harness Engineering

> 原文：[Harness engineering for coding agent users](https://martinfowler.com/articles/harness-engineering.html)
> 作者：Birgitta Böckeler (Thoughtworks)
> 日期：2026-04-02
> 备注：这是备忘录（2026-02-17）之后的完整版正式文章

---

*要让编码智能体在更少监督下工作，我们需要方法来增加对其结果的信心。作为软件工程师，我们对 AI 生成的代码有天然的信任屏障——LLM 是非确定性的，它们不了解我们的上下文，它们并不真正理解代码，而是以 token 为单位思考。本文探索一个心智模型，将上下文工程和 harness engineering 中的新兴概念整合起来，以建立这种信任。*

---

## 隐喻总有局限

"Harness"这个术语已经成为一种简写，意思是"AI 智能体中除模型之外的一切——Agent = Model + Harness"。然而，对于编码智能体来说，这个定义需要缩窄。Harness 包括内置组件（系统提示词、代码检索机制、编排系统）加上用户为特定用例构建的**外层 harness**。

> 侧边栏承认"在 harness 里面再套 harness 听起来不太对劲：'你试过把马具套在狗的里面吗？'"但如果这个隐喻确实有用，也就接受这种延伸了。

**图 1** 展示了三个同心圆：核心是模型，中间环是编码智能体构建者的 harness，最外层是编码智能体用户的 harness——每个边界上下文赋予"harness"不同的含义。

"一个精心构建的外层 harness 服务于两个目标：它增加了智能体第一次就做对的概率，同时提供一个反馈回路，在问题到达人类眼前之前尽可能多地自我纠正。"

## 前馈与反馈

方法涉及两种控制机制：

**引导器（前馈控制）** — 预判智能体的行为，在它行动**之前**引导它，增加首次尝试就获得好结果的概率。

**传感器（反馈控制）** — 在智能体行动**之后**进行观察，并启用自我纠正。"当传感器产生的信号被优化为适合 LLM 消费时，它们尤其强大——例如，自定义 linter 消息中包含自我纠正的指令——一种正面的提示词注入。"

文章提醒："如果分开使用，你要么得到一个不断重复同样错误的智能体（只有反馈），要么得到一个编码了规则但永远不知道规则是否起效的智能体（只有前馈）。"

## 计算性 vs 推理性

引导器和传感器有两种执行类型：

**计算性** — 确定性的、快速的、基于 CPU 的（测试、linter、类型检查器、结构分析）。毫秒到秒级运行，结果可靠。

**推理性** — 语义分析、AI 代码审查、LLM-as-judge。基于 GPU/NPU，更慢、更贵、更不确定。

"计算性引导器用确定性工具增加好结果的概率。计算性传感器足够便宜和快速，可以在每次变更时与智能体一起运行。"

**示例表：**

| 方向 | 计算性/推理性 | 示例 |
|------|-------------|------|
| 编码规范 | 前馈 | 推理性：AGENTS.md、Skills |
| 新项目引导指令 | 前馈 | 两者：带指令和引导脚本的 Skill |
| 代码变换 | 前馈 | 计算性：带 OpenRewrite 配方的工具 |
| 结构测试 | 反馈 | 计算性：pre-commit/编码智能体 hook 运行 ArchUnit 测试 |
| 审查指令 | 反馈 | 推理性：Skills |

## Harness Engineering 与 Context Engineering 的关系

> 侧边栏："Context engineering 为我们提供了将引导器和传感器提供给智能体的手段。为编码智能体设计用户 harness 是 context engineering 的一种特定形式。"

## 转向循环

"人类在其中的工作是通过迭代 harness 来**转向（steer）**智能体。每当一个问题反复出现时，前馈和反馈控制应该被改进，使这个问题在未来更不可能发生，甚至完全阻止它。"

"我们当然也可以使用 AI 来改进 harness"——智能体可以帮助编写结构测试、从模式中生成规则草稿、搭建自定义 linter 脚手架，或从代码库考古中创建操作指南。

## 时机：将质量左移

持续集成和交付团队根据成本、速度和关键性将"测试、检查和人工审查分布在开发时间线上"。原则是："越早发现问题，修复成本越低。"

关键问题包括：
- 什么在集成甚至提交创建之前运行（linter、快速测试套件、基础审查智能体）？
- 什么在集成后的管线中运行（变异测试、广泛的代码审查）？

**变更生命周期中的前馈与反馈：**

- **前馈示例：** LSP、architecture.md、/how-to-test skill、AGENTS.md、知识管理 MCP server
- **第一个自我纠正循环的反馈：** /code-review、eslint、semgrep、覆盖率、dep-cruiser
- **集成后管线传感器 + 昂贵选项：** /architecture-review skill、/detailed-review skill、变异测试

**持续漂移和健康传感器：**
- 持续监控渐进漂移（死代码检测、测试覆盖率质量分析、依赖扫描）
- 运行时反馈（SLO 退化建议、响应质量采样、日志异常标记）

## 规制维度

Harness 就像一个控制论调速器，将前馈和反馈结合起来，将代码库调节到期望状态。文章区分了三个维度：

### 可维护性 Harness

"我在本文中给出的几乎所有例子都是关于规制内部代码质量和可维护性的。这是目前最容易的 harness 类型，因为我们有大量可以利用的现成工具。"

作者将常见的编码智能体失败模式与此 harness 做了映射：

- **计算性传感器能可靠捕获的：** 结构性问题（重复代码、圈复杂度、缺失覆盖率、架构漂移、风格违规）——"这些便宜、经过验证、且是确定性的。"

- **LLM 部分能解决的：** 语义判断问题（语义重复代码、冗余测试、暴力修复、过度工程）——"但是昂贵且概率性的。不是每次提交都能跑。"

- **两者都无法可靠捕获的：** 高影响力问题："问题的误诊、过度工程和不必要的特性、误解的指令。"

### 架构适应度 Harness

"这组引导器和传感器定义和检查应用程序的架构特性。基本上就是：适应度函数（Fitness Functions）。"

示例包括：
- 描述性能要求的 Skills，配合性能测试反馈改善/退化情况
- 可观测性编码规范的 Skills，配合调试指令要求智能体反思日志质量

### 行为 Harness

"这是房间里的大象——我们如何引导和感知应用程序是否在功能上按我们需要的方式运行？"

当前方法包括：
- **前馈：** 功能规格说明（从简短提示词到多文件描述）
- **反馈：** AI 生成的测试套件验证、覆盖率分析、变异测试，加上人工测试

文章警告："这种方法对 AI 生成的测试寄予了太多信任，目前还不够好。"

## 可驾驭性（Harnessability）

不是每个代码库都同样适合被 harness。用强类型语言编写的代码库天然拥有类型检查作为传感器；可清晰定义的模块边界允许架构约束规则；像 Spring 这样的框架抽象掉了智能体甚至不需要操心的细节，因此隐式地增加了智能体的成功概率。没有这些属性，相应的控制就无从建立。

## 环境可供性（Ambient Affordances）

我的同事 Ned Letcher 用"环境可供性"这个术语来描述使智能体环境更具可驾驭性的属性："环境本身的结构属性使其对在其中运作的智能体来说是可读的、可导航的、可处理的。"

这在绿地项目和遗留系统中的表现截然不同。绿地团队可以从第一天就将可驾驭性融入设计——技术决策和架构选择决定了代码库的可治理程度。遗留系统团队，特别是那些积累了大量技术债的应用，面临更难的问题：**harness 最需要的地方恰恰是最难构建的地方。**

## Harness 模板

大多数企业有几种常见的服务拓扑覆盖了 80% 的需求——通过 API 暴露数据的业务服务、事件处理服务、数据仪表板。在许多成熟的工程组织中，这些拓扑已经被编码为服务模板。它们未来可能演变为 **harness 模板**：一组引导器和传感器的集合，将编码智能体约束在某个拓扑的结构、规范和技术栈上。团队可能开始部分基于已有什么 harness 来选择技术栈和结构。

## Ashby 定律

[Ashby 的必要多样性定律](https://en.wikipedia.org/wiki/Variety_(cybernetics)#Law_of_requisite_variety)指出："调节器必须至少拥有与被调节系统同等的多样性"，且"只能调节它拥有模型的东西。"基于 LLM 的编码智能体几乎可以生成任何东西，但**选定一个拓扑就缩窄了这个空间**，使构建全面的 harness 变得更可行。定义拓扑结构是一种多样性削减举措。

我们当然会面临与服务模板类似的挑战。团队一旦实例化模板，就开始与上游改进脱节。Harness 模板会面临同样的版本控制和贡献问题，甚至可能更严重——带有非确定性引导器和传感器的模板更难测试。

## 人类的角色

作为人类开发者，我们将自己的技能和经验作为隐性 harness 带到每一个代码库中。我们吸收了规范和最佳实践，我们体验过复杂性带来的认知痛苦，我们知道提交上有我们的名字。我们还携带着组织对齐——意识到团队试图达成什么、哪些技术债出于商业原因可以容忍、以及"好"在这个特定上下文中是什么样子。我们以人类的节奏小步前进，这创造了让经验被触发和应用的思考空间。

编码智能体没有这些：没有社会问责感，对 300 行函数没有审美上的厌恶，没有"我们这里不这么做"的直觉，也没有组织记忆。它不知道哪个规范是承重的、哪个只是习惯，也不知道技术上正确的解决方案是否符合团队正在努力做的事情。

**Harness 是将人类开发者经验所带来的价值外部化并显式化的尝试，但它只能做到一定程度。** 构建一个连贯的引导器、传感器和自我纠正循环系统是昂贵的，因此我们必须带着明确目标来排优先级：**一个好的 harness 不应以完全消除人类输入为目标，而应将人类输入引导到最重要的地方。**

## 起点——以及开放问题

这里描述的心智模型梳理了实践中已经在发生的技术，并帮助框定仍需解决的讨论。它的目标是将对话从特性层面提升——从 skills 和 MCP servers 上升到我们如何战略性地设计一个控制系统，使我们对智能体产出真正有信心。

当前来自业界讨论的示例：

- 一个 OpenAI 团队记录了他们的 harness 架构，包括"自定义 linter 和结构测试，以及定期运行的'垃圾回收'来扫描漂移"。他们的结论是："我们现在最困难的挑战集中在设计环境、反馈回路和控制系统上。"

- Stripe 的 minions 实现突出了"将反馈左移"的重要性，以 pre-push hooks 和将反馈传感器集成到智能体工作流中为特色。

- 变异测试和结构测试作为计算性反馈传感器，在被冷落后正在经历复兴。

- 关于在编码智能体中集成 LSP 和代码智能作为计算性前馈引导器的讨论越来越多。

- Thoughtworks 团队报告了用混合的计算性和推理性传感器来应对架构漂移，包括 API 质量改进和"清洁工军团"代码质量倡议。

**尚未解决的挑战：**
- 随着引导器和传感器增长，维持 harness 的连贯性，防止它们相互矛盾
- 当指令分歧时评估智能体的取舍
- 当传感器从未触发时，区分是高质量还是检测不足
- 开发类似代码覆盖率和变异测试的 harness 覆盖率评估方法
- 创建工具来将分散的前馈和反馈控制配置和推理为集成系统

构建外层 harness 是持续的工程实践，不是一次性配置。
