# 工具地图：复杂度维度 ↔ 工具杠杆

> 配套 [software-project-complexity-in-the-ai-era.md](../thinking/software-project-complexity-in-the-ai-era.md) 的6 维复杂度框架使用。

---

## 思路

每个工具的真正价值，不是它的功能列表，而是：

> **它能切实降低 AI 接管难度的哪一维？**

如果一个工具不能映射到任何一维复杂度的下降，它就不应该出现在这里。

### 分母：胶囊化能力（越高越好）

| 维度        | 工具杠杆候选                                | 实测笔记 |
| --------- | ------------------------------------- | ---- |
| **胶囊化能力** | Claude Code / Codex / Cursor / ... 等各大 harness 框架 | Harness 框架的核心价值——让 AI 在可控边界内完成任务，从而把上面六维的成本同时压下来 |

### 终局指标：重构 + 复刻

> 文章最后指出"重构能力 + 复刻能力"是复杂度的最终显影。一个工具是否值得收录，最终可以问：**它是否提升了项目的重构 / 复刻可行性？**

注：同一个工具可能横跨多维。**填表时按"主战场"放，次要贡献在工具页里展开。**

## 撰写新工具页的检查清单

- 是否回答了"它降低了哪几维复杂度"
- 是否给出至少一个**失效场景**（不是只写优点）
- 是否标注与其他已收录工具的搭配 / 冲突
- 是否说明"为什么是它而不是同类竞品"
